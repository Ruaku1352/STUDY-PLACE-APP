## 1. コンセプト
「勉強場所をコロコロ変える楽しさ」を味わいながら、計画的に週ごとの最低勉強ノルマを達成できるようにするアプリ。

**ターゲットユーザー**：休学中・長期休暇（夏休み等）など、毎日自由に使える時間がある人が勉強習慣をつける／維持するためのアプリ。**毎日が勉強日である前提**とし、休息日機能はあえて持たない（休みたい日はギブアップや終日の個人予定で対応）。
毎週、科目ごとのノルマ（目標勉強時間）と優先順位を設定すると、1日ごとの具体的なスケジュール（いつ・どこで・何を勉強するか）が自動生成される。

**デイリーガチャ形式（本アプリの独自性の核）**：
- 生成された週間プランは**ユーザーには事前に見せない**。
- **当日の朝、アプリを開いて「開封」すると、その日1日分のスケジュール（場所・科目・時間割）が初めて明かされる**。今日はどこで勉強することになるのか、開けるまで分からないガチャ的な楽しさを演出する。
- 開封時の逃げ道は2つだけ：**リロール（1日1回、場所を引き直して再生成）**と**ギブアップ（その日の分を通常表示にして手動編集可）**。
- 進捗ダッシュボード（ノルマ達成率）は常時閲覧可。**未開封日の場所・時間割だけが隠される**（ノルマの進み具合は隠さない）。

- 週の開始曜日：**月曜日**
- 1日の行動モデル：**朝、自宅を出発したら夜（21:00）まで自宅に戻らない**（外で勉強場所を渡り歩くスタイル）
- 自宅は勉強場所として扱わない

## 2. 技術スタック（TypeScript統一・将来のマルチユーザー化対応）
> 前提：まずは**自分専用**として完成させる（ハッカsoン前の成果物）。ただし**将来他人向けに公開できる構造**で最初から作る。

- **フレームワーク：Next.js（App Router）+ TypeScript**
  - 画面（React）とサーバー処理（API Routes / Server Actions）を1プロジェクトに統合
  - フロント・バックエンドを1言語（TS）で書き、型定義を共有する
- **認証：Auth.js（NextAuth）による Googleログイン**
  - Calendar連携で必要なGoogle OAuthをログインにも兼用する
  - 当面は自分のGoogleアカウントのみ許可（許可リスト方式）。将来は公開に切り替え
- **DB：Postgres（Neon の無料枠）+ Prisma（ORM）**
  - **全テーブルに `user_id` を持たせ、最初からマルチユーザー構造にする**（自分1人でも同じ構造で使う）
- **地図系API：Google Maps Platform（使用許可済み）**
  - **Routes API（Directions）**：移動時間の自動計算（`transit`＝電車＋`walking`）
  - **Places API（Place Details）**：営業時間・開館時間の自動取得（`opening_hours`）
- **カレンダー連携：Google Calendar API**（ログインと同じOAuthトークンを利用）
- **グラフ：Recharts**
- **ホスティング：Vercel**（Next.jsと相性最良。無料枠でOK）
- ⚠️ Google APIキー・OAuthシークレットは**サーバー側の環境変数**で保持し、クライアントに露出させない（課金キーのため）

### 将来のマルチユーザー公開時に必要になること（今はやらない・9章参照）
- Google OAuth 同意画面の本番審査（テストモード中は登録ユーザーのみ利用可能）
- API課金の管理（ユーザー数×API呼び出しが課金に直結するため、キャッシュ設計を最初から入れておくのはこのため）
- 利用規約・プライバシーポリシーの整備

## 3. 基本データ

### 3.1 科目 (Subject)
- 科目名
- 週間目標勉強時間（ノルマ、単位：分）※ユーザー本人が設定
- **優先順位**：毎週はじめにユーザーが手動で設定。時間が足りない週は優先順位の高い科目から割り当てる。
- **時間帯タグ**：`morning`（午前中に配置）/ `anytime`
  - 数学・アルゴリズム系は `morning`（デフォルト：出発〜12:00、変更可）。場所の指定はなし。

### 3.2 勉強場所 (Location)
- 場所名・住所（またはGoogle Place ID）
- 種別：`library`（滞在時間制限なし）/ `cafe`（最大滞在時間あり）等
- 最大滞在時間（分）※`cafe`等のみ。休憩・昼食を挟んでも滞在合計でカウント
- **営業時間**：Places API（`opening_hours`）から**自動取得**。取得不可の場所は手動入力にフォールバック。祝日・臨時休業はAPIに反映されない場合があるため、当日は手動リスケで対応。
- **移動時間：Google Routes API で自動計算**（手動登録は不要）
  - モード：`transit`（電車）＋`walking` を組み合わせ、最短の現実的な経路の所要時間を採用
  - 対象：自宅→各場所、場所→場所（前の滞在地点を起点に都度計算）
  - **キャッシュ必須**：同一ペア・同一時間帯の結果はDBにキャッシュし、API呼び出し回数と課金を抑える（有効期限：例として30日）
  - API取得に失敗した場合は、キャッシュ済みの過去値 → 手動入力値 の順にフォールバック

### 3.3 個人の予定 (FixedEvent)
- タイトル、開始・終了日時
- 場所（住所またはPlace ID。指定した場合、**予定終了後の移動はこの場所を起点に自動計算**）
- 場所未指定の予定は「その場に留まる」扱い（前後で移動なし）
- 種別（学校・バイト・その他）

### 3.4 ユーザー設定 (Settings)
- 自宅の住所（移動時間計算の起点）
- **起床時間／1日の開始時刻**（平日・休日で個別設定可）
- 午前（morning枠）の終了時刻（デフォルト12:00）
- 外での勉強の終了時刻（**21:00固定**。以降に勉強予定を入れない。帰宅の移動時間は考慮しない）

## 4. 週間プラン生成

### 4.1 入力
科目ノルマ・優先順位・時間帯タグ（週はじめに設定）／個人の予定／勉強場所リスト／ユーザー設定

### 4.2 出力
月〜日の7日分スケジュール。各日のタイムラインに「移動」「勉強（科目・場所）」「休憩」「昼食」「個人の予定」をブロック配置。

### 4.3 移動のルール
1. 1日の開始地点は自宅。最初の目的地への移動時間（Routes APIで自動計算）を先頭に確保。
2. 出発後は21:00まで自宅に戻らない。以降の移動はすべて**直前にいた場所を起点**にRoutes APIで計算。
3. 個人の予定に場所があれば、予定終了後はその場所を起点とする。
4. 帰宅の移動時間はスケジュールに含めない。

### 4.4 勉強ブロック配置のルール
1. **1つの場所での勉強ブロックの最小単位は1時間**。1時間未満の隙間には割り当てない。
2. **同じ場所は1日1回まで**（離れたら戻らない）。同じ場所に滞在したまま休憩・昼食を挟むのは連続滞在として可。
3. `morning` 科目は午前枠に優先配置。収まらない分は午後に回すが警告表示。
4. `cafe` 等は最大滞在時間を超えない（休憩込みの合計）。
5. 各場所は営業時間内のみ利用。21:00以降には配置しない。
6. 休憩・昼食は自動判断で適度に配置（固定ルールなし。極端な連続勉強・昼食抜きは避ける）。昼食は現在地付近で取り、移動時間は考慮しない。
7. **場所ローテーション**：「場所を変える楽しさ」を反映し、直近数日で使った場所の優先度を下げる。

### 4.5 ノルマ超過時の警告
- 週の全科目ノルマ合計と、実質スケジュール可能時間（予定・営業時間・21時制限・最小1時間ブロック制約を考慮）を比較。
- 超過する場合は**プラン生成時に警告**し、超過時間・割り当てきれない科目を明示。
- 自動でのノルマ変更はしない（調整はユーザー自身）。警告後も優先順位に従い可能な分は割り当てる。

## 5. デイリーガチャ（開封・リロール・リスケ）
1. **開封（Reveal）**：当日、ユーザーがアプリを開いて開封操作をすると、その日のスケジュールが表示される。開封前は場所・時間割を一切見せない（開封演出：カード/カプセルを開けるアニメーション程度でよい）。
2. **リロール（1日1回）**：開封結果が気に入らない場合、場所の組み合わせを引き直して当日分を再生成できる。ノルマ配分は維持し、場所とそれに伴う移動時間だけ変える。
3. **ギブアップ**：その日の分を通常のタイムライン表示に切り替え、手動編集（時間・場所変更・削除）を解放する。
4. **自動再計画**：「今日以降を再生成」機能。完了済みブロックの実績を保持し、**残りノルマ**を残りの日へ再割り当て。新規予定も反映。再生成された未来の日は再び未開封状態に戻る。
5. **Googleカレンダー同期**：アプリ発行イベントに識別ID（extended properties）を付与し、リロール・再計画時に古いイベントを自動で削除・更新。

---

## 6. 実績記録・進捗グラフ
1. 各勉強ブロックに「完了／一部完了（実施分数）／未実施」を記録。
2. **進捗ダッシュボード**（常時閲覧可。隠すのは未開封日の場所・時間割のみで、ノルマ進捗は隠さない）：
   - 科目別「今週のノルマ達成率」バー（例：数学 4.5h / 6h = 75%）
   - 週の経過日数に対する**理想ペース比較線**（どの科目が順調／遅れているか一目で分かる）
   - 遅れている科目は色分けで強調
   - （任意）過去数週間の達成率推移の折れ線グラフ
3. 実績データは自動再計画（5.4）の入力になる。

---

## 7. リマインド通知・カレンダー書き出し（ネタバレ防止）
- 週間プラン生成時、Googleカレンダーには**場所・科目を伏せたマスクイベント**（タイトル例：「🎁 スタディミッション」、時間帯のみ正しい）を書き出す。
- **当日開封したタイミングで、その日のイベントを実名（場所・科目入り）に更新**する。
- 各「勉強開始」イベントに**開始10分前の通知**を設定（Googleカレンダーの通知機能）。マスク状態でも時間帯は正しいので通知は機能する。
- 朝の開封を忘れないよう、1日の最初のイベント（出発前）にも通知を付ける。

## 8. 画面一覧
1. **設定画面**：自宅住所、起床時間（平日/休日）、午前枠終了時刻
2. **科目管理画面**：科目CRUD、週間ノルマ、時間帯タグ
3. **週はじめ優先順位設定画面**：ドラッグで優先順位設定 →「今週のプランを生成」（生成結果はこの時点では見せない）
4. **勉強場所管理画面**：場所CRUD。住所/Place ID入力→営業時間を自動取得（失敗時は手動入力）。種別・最大滞在時間の設定。移動時間は自動計算のため入力不要
5. **個人予定入力画面**：予定の登録（場所の紐づけ可）
6. **今日のページ（メイン画面）**：未開封なら開封ボタン（演出付き）→ 開封後は当日のタイムライン表示・実績入力・リロール（残1回）・ギブアップ・「今日以降を再計画」ボタン。ノルマ超過警告もここに表示
7. **進捗ダッシュボード**：達成率グラフ、理想ペース比較
8. **Googleカレンダー連携設定画面**：連携ON/OFF、同期状態

## 10. 実装上の注意（Claude Code向け）
- **Google APIの課金対策**：Routes API・Places APIの結果は必ずDB（Postgres）にキャッシュする（有効期限：例30日）。週間プラン生成1回で発生するAPI呼び出し数を見積もり、キャッシュヒット率を上げる設計にすること。キャッシュなしのループ呼び出しは厳禁。将来のマルチユーザー化ではこのキャッシュがコスト管理の生命線になる。
- **スケジューリングロジックは純粋関数として分離**し（移動時間・営業時間・予定などを入力に取り、ブロック配置を返すTSの関数群）、単体テスト（Vitest）を用意すること。
- **開発の順序**：①スケジューリングロジック（API未接続・ダミーデータ＋テスト）→ ②Google Maps/Places連携＋キャッシュ → ③タイムラインUI → ④Googleログイン＋Calendar連携 → ⑤進捗グラフ、の順で段階的に動くものを作る。
- **`user_id` を全テーブルに最初から入れる**。当面は自分のアカウント1つだが、クエリは常に `user_id` でスコープすること（将来の公開時に書き直しを発生させない）。
- Googleの認証情報は2種類ある：OAuth（ログイン＋Calendar）とAPIキー（Maps/Places）。混同しないこと。どちらもサーバー側の環境変数（`.env.local`）で管理し、リポジトリにコミットしない。
- 営業時間・移動時間が取得できない場合のフォールバック（手動入力UI）を必ず用意すること。
- **スマホでの利用を想定**したレスポンシブなUIにすること（外出先で実績入力・リスケ操作をするため）。
- **ハッカソン前の成果物として完成させる**ことがゴール。①〜⑤の各段階で「動く状態」を保ち、最後にVercelへデプロイして実運用（自分の1週間の勉強計画）まで通すこと。
